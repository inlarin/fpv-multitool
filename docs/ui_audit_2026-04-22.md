# UI audit: ELRS + CRSF tabs (2026-04-22)

## Measurements

| | ELRS tab | CRSF tab |
|---|---|---|
| HTML lines | 301 | 95 |
| Cards | 9 | 7 |
| Buttons | 21 | 7 |
| `<input>`s | 10 | 1 |
| `<select>`s | 6 | 0 |
| Warning blocks | 9 | 3 |
| JS functions called from HTML | 25 | 7 |

**Cognitive load**: 16 cards total across two tabs; 28 buttons; 9 scarlet warning blocks. Mostly Russian copy mixed with English technical terms (mode, DFU, stub, sync, OTADATA, chunk, slot).

## Mental model mismatch

The user doesn't distinguish between our two internal mechanisms:
- **"ELRS" tab path** = one-shot probe (acquire Port B → SLIP or CRSF frame → release).
- **"CRSF" tab path** = continuous `CRSFService` task (acquire Port B, stream telemetry for duration of tab).

Both talk CRSF to the RX on the same Port B pins at the same 420 000 baud. The distinction matters to US (for code locality) but makes no sense to the user. Evidence: CRSF tab has `Ping Device` + `Read Params` + `Bind` + `Reboot` + `Enter WiFi` — ELRS tab has `Probe RX` (does a DEVICE_PING ping) + `Load parameters` + `Enter binding` + `Enter stub flasher` + `Boot app0/1` + `Exit DFU`. **Every one of these ~10 actions has a functional cousin in the other tab.**

## Pain points — "I'm a new user"

### Tab-level

1. **Two tabs do overlapping things.** User sees "ELRS Flash" and "CRSF" in the header — which one do I open for «я хочу сменить пакет-рейт»? Neither label tells them. Answer happens to be "either works, one is better" — that's not UX, that's lore.
2. **No "where do I start" affordance.** No onboarding step, no highlighted "Probe first" badge. User scans 9 cards and picks one.
3. **Russian ↔ English mix**: "pct=27", "stage=Writing", "SILENT (WiFi/halted)", "Firmware Catalog", "сними питание с RX"... пёстрая ткань.

### ELRS tab

4. **9 cards visible by default, plus Advanced drawer**. Even after removing legacy dupes, primary + drawer total = 9 cards. All in scroll order, no grouping hint.
5. **`RX Configuration` card always visible** but its Load button is gated dim when not APP. A new user opens the tab, sees a disabled button, doesn't know why.
6. **`Controls` card mixes 5 unrelated actions**:
   - Enter binding  (app → binding mode)
   - Enter stub flasher  (app → SLIP stub)
   - Boot app0 / app1  (DFU → flip OTADATA + reboot)
   - Exit DFU → app  (DFU/stub → app)
   No visual grouping by precondition or outcome.
7. **"Architectural limits"** footer of Controls is three dense bullets explaining what CAN'T work. Good information — but the user first has to parse four tabs of dense Russian/English mix to read it. Belongs on hover / expandable, not as a wall of text.
8. **Flash Firmware's "Path" text** (`ROM DFU @115200 — full flash (bootloader + partitions + app)`) reads like a log line. Could be "via physical BOOT cable" or "auto" with a tooltip.
9. **Firmware Catalog's Version and Variant selects** ask the user to think about `FCC vs LBT` and `3.5.3 vs 3.6.3 vs 4.0.0`. Most users have no opinion. Default should be "latest stable + FCC" with a "Why does this matter?" expander.
10. **Advanced drawer is opaque** — no preview of what's inside. Users who NEED `Slot-targeted flash` or `Dump firmware` don't find it.

### CRSF tab

11. **Start/Stop is THE primary action** but is buried in card 1 as a small button next to an `Inverted` checkbox. A huge-button "Start live monitor" at top would signal "this tab is for monitoring".
12. **Link Stats / Channels / FC Telemetry cards render dashes with no explanation** when service is stopped. We have the "No TX link" hint now (good), but users don't know if zero data = service off vs no-link vs plate-issue.
13. **Device Info + Parameters + Commands** cards are 100% redundant with ELRS-tab equivalents. The only difference is `Parameters` uses the async cache (nicer) and `Commands` goes through `CRSFService` (richer — supports state-aware COMMAND params).
14. **`Enter WiFi Update Mode`** button exists with no clear warning that it's fragile (needs firmware to export "Enable WiFi" LUA param). Fails 404 silently-ish.

## Recommended redesign (scenario-driven)

### Option A — merge into one "Receiver" tab (radical, recommended)

Current two-tab split is for our code organisation, not user goals. One "Receiver" tab with three top-level sections:

```
╭─ Status ──────────────────────────────────────────────╮
│  [ Probe receiver ]   mode · ExpressLRS RX 3.5.3.0   │
│  Chip: ESP32-C3  ·  MAC: ..  ·  Boot slot: app0      │
│  (age badge, auto-refresh while tab open)            │
╰──────────────────────────────────────────────────────╯

╭─ What do you want to do? ─────────────────────────────╮
│                                                       │
│  Update firmware          →  [ Flash ... ]           │
│  Change settings          →  [ Load parameters ]     │
│  Watch link quality       →  [ Start live monitor ]  │
│  Pair with new handset    →  [ Enter binding ]       │
│  Reboot receiver          →  [ Reboot ]              │
│                                                       │
╰──────────────────────────────────────────────────────╯

(Sections expand inline when button clicked; only one open at a time)

╭─ ▸ Advanced (power-user) ─────────────────────────────╮
│   Slot-targeted flash  ·  OTADATA raw  ·             │
│   Dump firmware  ·  hardware.json export             │
│   Boot app0 / Boot app1 (direct OTADATA flip)        │
╰──────────────────────────────────────────────────────╯
```

Pros: matches user goals, not internal architecture. Single Port B lock
flow (auto-pause already there). ~50% fewer visible buttons.

Cons: bigger refactor (~200 LOC moved). Breaks existing link of users used to "CRSF tab".

### Option B — reorganise within existing two tabs (incremental, safer)

Keep two tabs, but:

**ELRS tab** becomes "Receiver · Setup" (flashing + config).
- Reorder: Status → Flash → Config → Commands → Advanced
- Split Controls card:
  - `Runtime commands` group (Bind, Enter stub) — visible when APP
  - `Bootloader commands` group (Boot app0, Boot app1, Exit DFU) — visible when DFU/stub
- Collapse legacy Advanced drawer behind a "Power tools" toggle with hover-preview

**CRSF tab** becomes "Receiver · Live" (live monitoring only).
- Remove: Device Info, Parameters, Commands cards (superseded by ELRS tab)
- Keep: Start/Stop (top, BIG), Link Stats, Channels, FC Telemetry
- Promote Start to a full-width primary button with "Connect to RX for live telemetry"
- No TX-link hint already good

Pros: less code churn, preserves user muscle memory.
Cons: two-tab ambiguity remains.

### Option C — keep current structure, polish only

- Add onboarding card at top of ELRS tab: "1. Probe RX · 2. Flash or Configure · 3. (optional) Monitor in CRSF tab"
- Group Controls buttons visually (background bands by precondition)
- Default version picker to 3.5.3 (known-good) with "Why?" tooltip
- Advanced drawer: add text preview "Contains: Slot-targeted flash, OTADATA raw, Dump"

Pros: minimal risk.
Cons: doesn't solve tab-ambiguity.

## Recommendation

**Go with Option B**. It addresses the biggest pain (CRSF tab duplicates) without a full rewrite. ~100 LOC of deletion + reordering in web_ui.cpp. CRSF tab becomes genuinely purpose-built for live telemetry (the unique thing it does well).

### Concrete steps (in priority)

1. **Delete from CRSF tab**: `Device Info`, `Parameters`, `Commands` cards (and associated JS `crsfPing`, `crsfReadParams`, `crsfBind`, `crsfReboot`, `crsfEnterWifi` + unused backend endpoints `/api/crsf/ping`, `/api/crsf/params`, `/api/crsf/read_params_blind`, `/api/crsf/params_list`, `/api/crsf/write`, `/api/crsf/bind`, `/api/crsf/reboot`, `/api/crsf/wifi`). Those users can do everything ELRS-tab side with better framing.

2. **Promote Start in CRSF tab**: full-width green button; label "Start live receiver monitor" (с переводом на русский). The inverted checkbox drops into a collapsed "Options" expander.

3. **Split ELRS `Controls` card into two** based on precondition: `Runtime commands` (visible when APP) and `Bootloader commands` (visible when DFU/STUB). Each shows only the buttons relevant to current mode.

4. **Architectural limits footer** → move to hover-tooltip on a "ⓘ What can't I do?" link. Reduces visual noise.

5. **Flash card default version**: preselect **3.5.3** (confirmed working on our test RX) with a small "Latest is 4.0.0" hint.

6. **Advanced drawer** gets a preview line: "Contains: Raw-offset flash · OTADATA debug · 4 MB dump · hardware.json export". Users know what's inside before opening.

7. **Onboarding banner** at top of ELRS tab (first visit, dismissable): "Connect receiver to Port B · Click Probe · Pick action below".

8. **Remove duplicate functionality Russians** translate or unify:
   - Keep Russian for action labels (more warm, closer to user)
   - Keep English for technical terms (DFU, app0, UART, CRSF — these are industry jargon users recognise)
   - Don't mix mid-sentence

## What to keep even in minimal redesign

- The status-badge color system (orange DFU, blue STUB, green APP, red SILENT) — visually informative.
- The "mode age" staleness timer — users see when to re-probe.
- Auto-probe after control actions — hides a round-trip.
- The P0 contention banners from 0e78177 — solve a real gotcha.
- Architectural-limits footer TEXT (just move to tooltip, don't delete — it's true and helpful).

## Out of scope here

- Translating entire UI to single language (huge, political).
- Mobile layout (cards stack fine; buttons could be bigger).
- Theming / dark-mode already works.
- i18n framework (overkill).
