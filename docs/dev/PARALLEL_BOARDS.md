# Parallel Board Development — Self-Instructions

This document is the rulebook for working on the FPV MultiTool's two
target boards simultaneously without breaking either. Read at the start
of every session that touches code outside a single board's UI directory.

The goal: **shared code stays shared, board code stays board-local, and
neither board falls behind the other on features.**

---

## 1. The two boards

| | **Waveshare ESP32-S3-LCD-1.47B** | **WT32-SC01 Plus** |
|---|---|---|
| Codename in code | `esp32s3` (the original) | `wt32_sc01_plus` |
| MCU | ESP32-S3-WROOM, R8 (OPI PSRAM) | ESP32-S3-WROOM, R2 (QSPI PSRAM) |
| Display | ST7789 172×320 SPI | ST7796 320×480 i80 8-bit |
| Touch | none | FT6336 I2C |
| Local UI | RGB LED + BOOT button menu | LVGL springboard + touchscreen |
| Display lib | `Arduino_GFX` 1.5.6 | `LovyanGFX` 1.2.x + `lvgl` 9.2.2 (pinned) |
| SD card | SDMMC 4-bit, optional | SDMMC 1-bit, mounted on boot |
| Battery | onboard LiPo + ADC | external power only |
| `psram_type` | `qio_opi` | `qio_qspi` |
| `ARDUINO_USB_MODE` | `0` (TinyUSB composite) | `1` (standard CDC) |
| `ARDUINO_USB_CDC_ON_BOOT` | `0` | `1` |
| Pin map | `pin_config.h` (default) | `pin_config_sc01_plus.h` (via `BOARD_WT32_SC01_PLUS`) |
| MAC | `3C:DC:75:6E:CE:A8` | `88:56:A6:80:EB:48` |
| Default COM port | `COM8` | `COM13` |
| IP (current LAN) | `192.168.32.50` | `192.168.32.51` |

**Both boards expose USB ID `303A:1001`.** Flashing the wrong build to
the wrong board produces an unbootable image (incompatible PSRAM
controllers). **Always verify MAC before any direct upload.** OTA is
safe — it goes by IP.

## 4. Workflow protocols

### 4.1 Adding a feature shared between both boards

A feature is "shared" when it touches Port B subsystems (battery,
servo, motor, CRSF, RCSniffer), web routes, or the safety net.

1. Code lives under `src/<feature>/`. **Never** `src/board/<X>/`.
2. Feature code talks to hardware ONLY via:
   * `PinPort::acquire(PORT_B, mode, owner)` — never hard-coded GPIO
     numbers. Pin map differs between boards; PinPort respects it.
   * `BoardDisplay` (only on SC01 Plus screens — Waveshare draws
     directly via Arduino_GFX).
   * `WebState::Lock` for any cross-task shared state.
3. Web route, if any, goes under `src/web/http/routes_<area>.cpp`.
4. **Build both envs before committing:**
   ```powershell
   pio run -e esp32s3
   pio run -e wt32_sc01_plus
   ```
   If either fails, the feature isn't done.
5. Test on whichever board is in hand. If the feature is critical
   (motor, battery), test on **both** before merging.

### 4.2 Adding an SC01-Plus-only LVGL screen

1. Source file: `src/board/wt32_sc01_plus/ui/screen_<name>.cpp`.
   Mandatory contents per [the back-button audit](../../CLAUDE.md):
   * `static lv_timer_t *s_tick = nullptr;` (if the screen needs a
     refresh tick)
   * `static void <name>Cleanup(lv_event_t *)` that
     `lv_timer_delete(s_tick)`s and nulls every static widget pointer
   * `lv_obj_add_event_cb(panel, <name>Cleanup, LV_EVENT_DELETE, nullptr)`
     in the builder
   * **Never** mass-iterate `lv_timer_get_next` — kills LVGL internals
2. Public entry: `void buildXxx(lv_obj_t *panel)` declared in
   `src/board/wt32_sc01_plus/ui/screens.h`.
3. Wire into the springboard: add a `SectionSpec` row in
   `src/board/wt32_sc01_plus/ui/board_app.cpp::SECTIONS[]`.
4. Long-running ops in click handlers must NOT block loopTask. Anything
   over ~30 ms (SD_MMC.begin, WiFi.begin, RCSniffer::autoDetect, large
   I/O) must go to a state-machine driven by `lv_timer_create` one-shots
   or a dedicated FreeRTOS task. See `screen_sniff.cpp` detect state
   machine for the canonical pattern.
5. AsyncWebServer routes that touch any `lv_*` API MUST grab
   `BoardApp::lvLock()` first and release in RAII guard. Bail with 503
   if lock times out.
6. Build wt32_sc01_plus env. Smoke-test via
   `scripts/smoke_test.ps1` — walks all sections + back, asserts no
   reboot.

### 4.3 Adding a Waveshare-only feature

1. Source goes under `src/board/wsh_s3_lcd_147b/`:
   * Hardware abstractions (display, button, status_led, usb2ttl) at
     the board root.
   * Screens / UI flows under `src/board/wsh_s3_lcd_147b/ui/`.
2. The SC01 Plus env's single `-<board/wsh_s3_lcd_147b/>` exclude
   already covers any new file you drop in there — no
   `build_src_filter` edits needed.
3. Build both envs to confirm.

### 4.4 Modifying shared code

1. Read the consumers on both sides before editing. `src/web/web_state.h`
   has cross-task semantics; `src/core/pin_port.cpp` is the only
   reachable hardware mux.
2. **Build both envs.** If shared code change breaks either, it's
   not shared code anymore.
3. If the change has runtime behaviour (e.g. PinPort acquire timing),
   smoke-test on both boards. Use both `scripts/smoke_test.ps1` (SC01
   Plus) and the Waveshare web UI (open `/` and click through tabs).

### 4.5 Releasing a tagged version

1. Make sure both envs build.
2. Tag from `main`:
   ```powershell
   git tag -a v0.X.Y -m "..."
   ```
3. `inject_version.py` bakes the tag into `src/core/build_info.h` on
   the next `pio run`. The status bar (SC01 Plus) shows it.
4. Patch releases (v0.X.Y bumps): unrelated bug fixes can ship in the
   same tag. Minor releases (v0.X.0): one or more major features.

## 5. Pre-flash safety checklist

Before `pio run -t upload` or USB flash:

1. **Identify which board you have.** Read its MAC:
   ```
   esptool.py --port COM<N> chip_id
   ```
   * `3C:DC:75:6E:CE:A8` → use `[env:esp32s3]` only
   * `88:56:A6:80:EB:48` → use `[env:wt32_sc01_plus]` family only
2. **Match env to MAC.** Swapping these turns the board into a brick
   until USB recovery (the boot ROM expects the right PSRAM controller
   on first init).
3. **Prefer OTA over USB.** OTA goes by IP (`scripts/ota_flash.sh
   <env>`) — no USB-side board confusion possible. Reserve USB for
   first-time provisioning, partition changes, and recovery.
4. **Never `--no-verify` and never bypass `--upload-port`.** The
   pio auto-detect will silently pick whichever board enumerated first.

## 6. Anti-patterns — never do these

* **Hard-coded GPIO numbers in `src/<feature>/`.** Always go through
  `PinPort::xxx_pin()` helpers. The two boards have different pin maps;
  `pin_config.h` dispatches but only if features actually use the
  symbolic names.
* **`lv_*` calls from any web route handler without `BoardApp::lvLock`.**
  Routes run on AsyncTCP task. Touching LVGL there is a data race
  with `lv_timer_handler` on loopTask. See commit `d2620aa`.
* **`portMAX_DELAY` on any mutex reachable from AsyncTCP task.** Bound
  every wait. AsyncTCP is subscribed to TWDT; an indefinite wait
  silently bricks the device.
* **`SD_MMC.begin`, `WiFi.begin`, `Wire1.begin`, `RCSniffer::autoDetect`
  inline from a click handler.** All can block hundreds of ms to
  multiple seconds. Defer to boot path or a state machine.
* **`lv_obj_clean` / `lv_obj_del` from inside `LV_EVENT_DELETE` /
  `DELETED` handlers.** Use `lv_async_call` or `lv_obj_delete_async`.
  See commit `debcd3b` and `d7bd8c7`.
* **Centralised "kill all lv_timers" logic.** LVGL's own indev/display
  refresh timers live on the same global list. Each screen owns and
  cleans up only its own `s_tick`.
* **Editing a Waveshare-only file without adding it to the SC01 Plus
  exclusion list.** Build will fail or, worse, link a Waveshare-shaped
  symbol into the SC01 Plus binary.
* **Adding a new build dependency to one env without checking the
  other.** Pinned versions matter — see `lvgl@9.2.2` exact pin
  ([reference_lvgl_9_5_glyph_regression.md](../../../../.claude/projects/c--Users-inlar-PycharmProjects-esp32/memory/reference_lvgl_9_5_glyph_regression.md)
  in user memory).
* **`gpio_reset_pin` skipped when transitioning a Port B pin between
  modes.** Wire1.end + ledcAttach on the same pin without it = silent
  failure. See `servo_pwm.cpp`.
* **LEDC channel auto-allocation on the SC01 Plus.** LovyanGFX backlight
  uses channel 7 / timer 3. New PWM consumers must `ledcAttachChannel`
  to a channel on a different timer (0, 1, or 2). See commit `edc72f8`.

## 8. Boards-feature parity matrix (audit before any release)

Update this when adding / removing features.

| Feature | Waveshare | SC01 Plus | Source |
|---|:-:|:-:|---|
| Servo bench tester | ✅ web + local screen | ✅ web + LVGL | shared `servo/` |
| Motor (DShot) | ✅ web + local screen | ✅ web + LVGL | shared `motor/` |
| Battery quick read | ✅ web | ✅ web + LVGL | shared `battery/` |
| Battery service ops (unseal/clearPF/cycles) | ✅ web | ❌ web only | shared `battery/` |
| ELRS RX link monitor | ✅ web | ✅ web + LVGL | shared `crsf/` |
| ELRS firmware flash | ✅ web | ❌ web only (Phase 3 LVGL) | shared `web/http/routes_flash.cpp` |
| RC Sniff (SBUS/iBus/PPM) | ✅ web | ✅ web + LVGL | shared `rc_sniffer/` |
| SD card catalog browser | ❌ | ✅ LVGL Phase 1 | SC01 Plus has SD wired in boot |
| SD card catalog flasher | ❌ | ⏳ Phase 3 | shared (when ready) |
| OTA + safety net | ✅ | ✅ | shared `core/safety.cpp` |
| Factory + safeboot recovery | ❌ | ✅ | SC01 Plus only — Waveshare has no factory partition |
| Outbound health beacon | ✅ | ✅ | shared `core/safety.cpp` |
| Status bar version display | ❌ | ✅ | LVGL springboard, SC01 only |
| Touch calibration | n/a | ✅ | SC01 Plus only |
| BOOT button menu | ✅ | n/a | Waveshare only |

**Rule:** when a feature is shared (last column says `shared/`), both
boards' UIs that surface it should advance together. If you ship the
SC01 Plus LVGL screen for X without keeping the Waveshare web tab
working, you've broken parity. Document any deferral with a date in
`CLAUDE.md` under `Pending architectural work`.
